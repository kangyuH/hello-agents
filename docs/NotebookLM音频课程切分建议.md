# NotebookLM 音频课程切分建议

本文档用于将 `docs` 目录中的 HelloAgents 教程拆分为更适合 NotebookLM 生成音频课程的学习单元。

由于单段音频能承载的内容有限，拆分时不建议机械按“章”处理，而应按“一个清晰认知单元 = 一节音频课”的原则组织。每节课最好只围绕一个核心概念、一种工程范式、一个平台实践、一个工具模块，或一个完整项目阶段展开。

## 一、切分原则

### 1. 每节课只解决一个主要问题

例如：

- “什么是智能体”适合单独讲。
- “ReAct 的运行机制和编码实现”适合单独讲。
- “BFCL 工具调用评估”适合单独讲。
- “第九章上下文工程”内容过密，不适合整章生成一节音频。

### 2. 理论章可以适度合并，工程章需要细切

第 1-3 章偏入门和背景，可以按章内主题合并。第 7-12 章涉及框架、记忆、上下文、协议、训练和评估，是全书知识密度最高的部分，建议细拆。

### 3. 项目章按工程模块拆

第 13-15 章是完整项目案例，不建议整章生成音频。更适合按以下结构拆：

- 项目架构
- 后端与智能体设计
- 工具或协议集成
- 前端或交互实现
- 项目复盘与扩展

### 4. 每节课建议控制在 15-25 分钟

如果 NotebookLM 生成的内容明显过短，说明素材过少或提示词不够具体；如果内容过泛，说明输入材料过多，需要继续拆分。

## 三、NotebookLM 音频生成提示词

下面的提示词适合在 NotebookLM 只能看到当前小节资料、无法看到全书全局结构时使用。建议每次生成音频前，把对应小节材料和提示词一起放入 NotebookLM。

### 1. 通用提示词

```text
你正在基于当前资料生成一节 HelloAgents 音频课程。

重要背景：
- 这是一门从智能体基础、经典范式、框架开发、记忆检索、上下文工程、通信协议、训练评估，到完整项目实战的系统课程。
- 你当前只能看到本节课资料，无法看到完整课程全局。因此请不要假设听众已经掌握本节之外的细节。
- 但也不要把课程讲成泛泛而谈的百科介绍。请紧扣当前资料，围绕本节主题展开。

请按“音频课程”而不是“文章摘要”的方式生成内容：
1. 开场先用 1-2 分钟说明本节要解决的核心问题，以及它在智能体学习路线中的位置。
2. 然后提炼 3-5 个核心知识点，每个知识点都要解释“是什么、为什么重要、如何在工程中使用”。
3. 如果资料中有代码，请不要逐行朗读代码，而是解释代码背后的设计思路、关键流程、输入输出、容易出错的地方。
4. 如果资料中有平台或项目操作，请按“目标、步骤、关键配置、验证方式、常见问题”来讲。
5. 如果资料中出现前置概念，请用简短语言补齐背景，但不要长篇展开无关内容。
6. 如果资料中出现后续章节内容，请只做轻量预告，不要替后续章节展开。
7. 结尾请总结本节的 3 个核心 takeaway，并给出一个适合学习者课后思考或动手练习的小任务。

语气要求：
- 使用中文。
- 像一位耐心的工程课程讲师，而不是念稿机器人。
- 多用“我们先看”“这里真正重要的是”“工程上要注意”这类引导语。
- 避免堆砌术语。每个重要术语第一次出现时都要解释。
- 不要把参考文献、习题、图片路径逐条念出来，除非它们对理解本节很关键。
```

### 2. 入门理论课提示词

适合第 1-3 章，以及每章的概念导入部分。

```text
请把当前资料改写成一节适合初学者收听的入门音频课。

本节目标不是覆盖所有细节，而是帮助学习者建立正确的心智模型。请优先讲清：
1. 这个概念解决什么问题。
2. 它和传统方法有什么区别。
3. 它为什么会在智能体系统中变得重要。
4. 学习者最容易误解的地方是什么。
5. 后续学习中应该带着哪些问题继续看。

如果资料中有历史脉络，请按“问题出现 -> 旧方案 -> 旧方案限制 -> 新范式出现”的方式讲。
如果资料中有公式或架构，请用直观类比解释，不要让音频变成公式朗读。
```

### 3. 编程实战课提示词

适合第 4、7、8、9、10、11、12 章中的代码实现部分。

```text
请把当前资料生成一节偏工程实战的音频课程。

讲解重点：
1. 先说明本节要实现的功能，以及它在 Agent 系统中的位置。
2. 再按模块解释代码结构，而不是逐行朗读代码。
3. 对每个关键类、函数或工具，请说明它的职责、输入、输出，以及它和其他模块的关系。
4. 对复杂流程，请用“第一步、第二步、第三步”的方式讲清数据如何流动。
5. 请特别指出容易出错的地方，比如环境变量、API Key、异步调用、工具注册、模型输出格式、状态管理、依赖安装等。
6. 结尾给出一个最小可运行理解路径：学习者应该先运行什么，再观察什么，再尝试改什么。

不要逐字朗读大段代码。对于代码，只解释关键设计、执行流程和工程注意事项。
```

### 4. 平台操作课提示词

适合第 5 章低代码平台，以及项目中的工具配置部分。

```text
请把当前资料生成一节平台操作型音频课。

请按以下结构讲：
1. 这个平台或工具适合解决什么问题。
2. 本节案例最终要做出什么效果。
3. 搭建过程分成哪些关键阶段。
4. 每个阶段最重要的配置是什么。
5. 哪些地方最容易配置错。
6. 这个平台相比写代码的优势和局限是什么。

请避免把界面截图或按钮路径机械念出来。要把操作背后的意图讲清楚，让学习者即使界面变化，也知道自己在配置什么。
```

### 5. 项目案例课提示词

适合第 13-15 章。

```text
请把当前资料生成一节项目案例型音频课。

这节课要帮助学习者理解一个真实 Agent 应用是如何拆成工程模块的。请重点讲：
1. 项目要解决的用户问题是什么。
2. 系统由哪些模块组成，每个模块负责什么。
3. Agent、工具、后端服务、前端界面之间如何协作。
4. 当前资料对应的是项目中的哪一层。
5. 当前这一层的关键实现决策是什么。
6. 如果学习者要复现这个项目，应该优先跑通哪条主流程。

请不要把项目讲成普通产品介绍。重点放在工程架构、数据流、模块边界、实现取舍和可复用经验。
```

### 6. 训练与评估课提示词

适合第 11-12 章。

```text
请把当前资料生成一节训练或评估主题的音频课。

请特别注意：
1. 先解释训练或评估要解决的核心问题。
2. 明确输入是什么、过程是什么、输出是什么。
3. 如果涉及数据集、奖励函数、指标或排行榜，请解释它们衡量的能力边界。
4. 如果涉及训练方法，请讲清它和普通提示工程、普通微调之间的区别。
5. 如果涉及评估方法，请讲清它能发现什么问题，不能发现什么问题。
6. 对工程实践中的成本、风险和常见失败模式做提醒。
```

## 五、针对 NotebookLM 的额外注意事项

### 1. 不要一次喂完整大章

第 7-12 章尤其不建议整章输入。它们单章内容量大、代码多、概念密集，NotebookLM 容易生成表面化总结。

### 2. 给它明确的“本节边界”

NotebookLM 不知道你整个课程怎么安排，所以要明确告诉它：

- 本节只讲什么。
- 哪些内容只做背景补充。
- 哪些内容留到后续课程。

### 3. 对代码章节要禁止逐行朗读

代码章节最容易生成“代码播报”。提示词中一定要强调：

- 不要逐行解释。
- 讲模块职责。
- 讲数据流。
- 讲关键设计。
- 讲常见错误。

### 4. 对项目章节要强调工程视角

项目章节不要生成产品宣传式内容，应要求它讲：

- 架构设计
- 模块边界
- 数据如何流动
- Agent 如何调用工具
- 前后端如何协作
- 哪些实现可以复用到其他项目

### 5. 每节课结尾都要求 takeaway 和练习

这样可以让每段音频更像课程，而不是普通摘要。

建议固定结尾结构：

```text
请在结尾输出：
1. 本节三个核心 takeaway。
2. 一个 10-20 分钟可以完成的课后练习。
3. 一个进入下一节课前应该思考的问题。
```
